The basic usage of the package involves the “nowcast” function, which fits a single nowcasting model to the data.

The formula interface is used to specify the response variable and covariates, and lags can be specified using the “lag” function within the formula. The function converts the data into a standard format across all models so that a single “multidadnow” object can be used to store the results of multiple models, and the same plotting and evaluation functions can be used regardless of the underlying model.

The add_nowcast() interface allows for new formulas to be specified when adding new models. Notice that a message will be displayed if no formula is specified, in which case the formula from the first registered model will be used.

dadnow2 <- nowcast(
  formula = dad ~ lag(cnisp, 3) + lag(ptsos, 3),
  data = df, 
  model = "ar", 
  date_col = "date", 
  params = list(p = 1)
)
#> Model: ar

dadnow2 <- add_nowcast(dadnow2, model = "ar", params = list(p = 2)) |>

There’s also an autoplot() method for “multidadnow” objects, which plots the nowcast against the observed data. By default, it plots all training data in the “multidadnow” object, but the last_n argument can be used to specify how many of the most recent observations to plot.

The function uses the same data as the output of get_data(), so the data can be extracted from the “multidadnow” object and used in the plot.

The package includes a “mechanistic” model, which can be seen in vignette("Mechanistic_Model", package = "dadnowcast"). The model takes the parameters sc and sp and requires a method (“normal”, “poisson”, or “negbinom”) to specify the distribution of the error term. Because it works slightly differently than the other models, it is added to a “multidadnow” object using the add_mechanistic function, which takes the same formula interface as nowcast but with different parameters. The mechanistic model can be added alongside the other models, and the same plotting and evaluation functions can be used.

Note that the mechanistic assumption is sensitive to the values of sc and sp, and there will often be warnings about lack of convergence in the optimization process.

mech <- nowcast(
  model = "mechanistic",
  dad ~ cnisp + ptsos + npos, df, date_col = "date",
  params = list(sc = 0.2, sp = 0.3, method = "poisson")
)

Evaluation Metrics

The evaluation metrics come from the dadnowcast:::enbpi() function, which uses Ensemble batch prediction intervals to calculate prediction intervals and prediction metrics. This works by creating many different “batches” (which are contiguous sets of data, i.e., using all data in a time interval), each with the same number of training observations; fitting a model to the batch; and then predicting \(k\) steps ahead (\(k\) is chosen based on the amount of trailing NAs in DAD data, i.e., the number of nowcasts to be made). For example, if there are 10 points in the data to be nowcast and you choose batches=40 with training_window=NULL, then the function will create 40 batches, each with 60% of the training data, fit a model, then calculate the 10-step ahead predictions from each model. The batches all have the same size but a random starting index, meaning that we get data without time jumps.

From each batch, we have a set of predictions compared to actual values. The standard deviation of the 1-step ahead prediction can be calculated from the batches, then the standard deviation of the 2-step ahead, etc. This gives us an empirical measure of the prediction intervals for our particular nowcasting scenario.

This is similar to k-fold cross-validation, but targeted to a nowcasting scenario. With k-fold CV, the data into k folds and we predict all of the data from each fold. This does not take the ordering of the data into account, and the evaluation metrics are based on the amounts of data in each fold (for example, in 10-fold CV, each fold has 1/10 of the data). EnBPI takes the ordering into account and also finds metrics (RMSE, MAE, and MRE) based on exactly the sort of predictions that we are making.

  • More batches means more accurate prediction intervals, but also more computationally intensive.
  • The default train_window uses 60% of the training data for each batch, leaving space for a variety of intervals while also using enough of the data for training.
    • To see why, consider the case where we have 100 data points, 5-step ahead predictions, and we use 95% of the data in each batch. Since we need space to do 5-step ahead predictions, every single batch is going to be the first 95 data points! If we used 60% instead, then the batches will have starting indices anywhere between 0 and 35, so we have a greater variety of k-step ahead predictions.
    • If we used 5% of the data in each batch, we would have lots of variety in the nowcasts, but we would be training on only 5 data points each time, which is not a good reflection of the amount of data that we actually have.
  • The level is the prediction interval level. The prediction intervals are calculated based on an asymptotic normality assumption: the standard error (\(SE\)) is caclulated empirically, then the interval is calculated as \(\hat y \pm z_{(1 - \texttt{level})/2} SE\).

The parameters for EnBPI can be specified in the nowcast function, then add_nowcast() will read those parameters and use them for any additional nowcasts. The defaults are shown below in dadnow4. If two sets of multidadnow objects have different values for batches, train_window, or level, then combine_nowcasts() will fail since the results are not comparable.

system.time(
  dadnow4 <- nowcast(
    dad ~ lag(cnisp, 3) + lag(ptsos, 3),
    df, model = "ar", date_col = "date", params = list(p = 1),
    batches = 40, train_window = NULL, level = 0.95
  )
) |> print()

In the lm above, you’ll notice that the lag terms were expanded into the model matrix. If you want to use other data to make predictions, you can use the prep_newdata() function to create a data frame with the data that lm expects. This can be used to, say, make predictions for one province based on a model fit to a different province.


new_data <- prep_newdata(
  formula = get_formula(dadnow2, "lm_f1"),
  data = df,
  date_col = "date",
  interpolate = TRUE
)

# Note: the three NA values are because of the lag terms.
